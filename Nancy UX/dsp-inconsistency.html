Ads UI Inconsistency


Inconsistency within a single web application can be critical for user experience.
Over time, new pages, design work or designers / engineers often don't work within the design system or style guide, leading to the different inconsistencies

The Ads Platform UI Inconsistencies project is meant to clean up design inconsistencies in the Ads Platform UI via a quick and flexible process outside of quarterly Ads Platform product prioritization.

My contribution includes:

  • Led the Inconsistency project successfully
  • Applied the Unified Design System on the main components across the Ads app
  • Provided the sheet for elements, including: priority, level of effort, order and importance
  • Followed with the Engineering, UX, PMs making sure it was released successfully
Why does Ads Platform UI have Inconsistencies?
  • Ads Platform UI Took over from brightroll
  • Different Ember upgrades
  • Different rebrandings
  • Different teams (development styles - design styles - PM styles)
  • Very big App, many features.
  • Clean ups - Restructuring - Atomic - Grid unit (Different coding styles)
Ads Platform UI Inconsistencies categories:

Having different meetings with different stakeholders (UX-ENG-PM) we came up with below 4 categories for design elements:

  • Ready to fix
  • Will wait for Rize (Design System)
  • Will happen in Rize but we can do it now
    • Rize might take long and we need it now
    • It is not heavily used so we can apply the new UDS to make it reusable
    • General component, centered code, big effect with less code
  • Layout-Grid, and clean up code that will make the integration with Rize easier
